## 🔴 不可变约束（铁律 — 禁止事项）

> 以下规则在任何情况下都不可违反。Implement 阶段自动排除违反以下规则的方案。

### 安全

| # | 规则 | 理由 |
|---|------|------|
| SEC-01 | 任何 Token/Key/Secret 不得硬编码在源代码中 | 防止泄露 |
| SEC-02 | 用户输入必须做 SQL 注入/XSS 防护 | 安全基线 |
| SEC-03 | 敏感配置通过环境变量注入 | 12-Factor App |

### 架构

| # | 规则 | 理由 |
|---|------|------|
| ARCH-01 | 禁止循环依赖 | 维护性 |
| ARCH-02 | 数据库迁移不可逆（只能 forward，不能 rollback） | 数据安全 |
| ARCH-03 | API 必须版本化（v1/v2） | 兼容性 |

### 代码

| # | 规则 | 理由 |
|---|------|------|
| CODE-01 | 所有公开函数必须有 docstring | 可读性 |
| CODE-02 | 新功能必须有测试（覆盖率 >= 80%） | 质量保证 |
| CODE-03 | 禁止 `print` 调试，使用 logging 模块 | 生产规范 |

### 部署

| # | 规则 | 理由 |
|---|------|------|
| DEPLOY-01 | 禁止直接在生产环境修改代码 | 变更管理 |
| DEPLOY-02 | 部署前必须通过所有测试 | 质量门禁 |

## 📐 接口约定

### REST API 格式

```json
// 成功
{ "code": 0, "data": {...}, "message": "ok" }

// 失败
{ "code": 1001, "data": null, "message": "参数错误: xxx" }
```

### 错误码规范

| 范围 | 含义 |
|------|------|
| 0 | 成功 |
| 1000-1999 | 参数错误 |
| 2000-2999 | 业务逻辑错误 |
| 3000-3999 | 外部服务错误 |
| 9000-9999 | 系统内部错误 |

## 📋 使用说明

1. 复制此模板到项目根目录，重命名为 `.hermes-constitution.md`
2. 根据项目实际情况修改各章节
3. SPEC-DRIVEN-DEV 管道启动时会自动搜索并加载此文件
4. 搜索顺序：项目根目录 → 父目录 → `~/.hermes/`（全局）
